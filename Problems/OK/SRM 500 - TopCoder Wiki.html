Single Round Match 500
Saturday, March 19th, 2011

Match summary

It's been a long time since SRM 1. In that SRM, the 81 participants were asked to perform a compound interest calculation, enumerate fractions with denominators up to 12, and investigate the partitions of a number into three relatively prime numbers. All these problems were solvable by brute force or simple simulation. The coders represented only two countries: Canada and the USA. Sadly, none of these initial stalwarts are actively competing today.

But things have changed a lot since that first match! SRM 500 comprised 2,100 participants, and together they represented more than eighty different countries (including, perhaps improbably, Antarctica). The problems, set by Chmel_Tolstiy, were interesting, but were also rather tricky. In fact, more than half of the Division 1 contestants did not manage to solve any of the problems correctly in the allotted time. Even the easiest Division 2 problem needed some non-trivial analysis, and the simulation problem common to both divisions was far more complicated and had a much longer description than any of the SRM 1 problems.

But some things haven't changed. The competition is still fierce. The problem quality remains high, and nearly every match runs without a hitch. This is due to the hard work of the writers, testers and administrators behind the scenes, which contribute to making SRMs such a wonderful experience for the participants.

TopCoder celebrated SRM 500 by commisioning the design of a special logo and a series of member articles reminiscing on the history of SRMs. It also generously awarded cash prizes, patches and commemorative T-shirts to many lucky contestants. In Division 1, the winners included only two people who solved all three problems: the indomitable Petr and the up-and-coming tourist. In Division 2, no one managed to solve all three problems, but I_L_P scored a convincing victory with well over a thousand points. Congratulations to them, and to all the other prize-winners.

In ten years time, we might reach SRM 1000. Assuming similar growth rates to those in the previous decade, fifty thousand competitors will participate, and they will hail from 3,200 different regions (including, perhaps improbably, Mars). The competitors will have to solve three challenges involving quantum computing, distributed algorithms, and polynomial-time solutions for NP-complete problems. Well, perhaps. You never know. But, in the meantime, here's to another decade of exciting SRMs!

The Problems

SRMCards
Used as: Division Two - Level One:
Value 250
Submission Rate 955 / 1094 (87.29%)
Success Rate 852 / 955 (89.21%)
High Score damonli for 249.96 points (0 mins 21 secs)
Average Score 193.84 (for 852 correct submissions)

There's a bunch of cards with numbers. In each turn, you must pick a card (call its number A), and also remove any cards with the numbers A - 1 or A + 1. What is the maximum number of turns possible?

First, notice that we can group the cards into maximal contiguous sequences. For example, given the set {1, 8, 9, 11, 12, 13, 99, 100, 101, 102}, we can partition it into {1}, {8, 9}, {11, 12, 13} and {99, 100, 101, 102}. This is a sensible thing to do because picking a card only affects its two immediate neighbors - so the cards we pick from each partition can't possibly affect the cards we pick from another partition.

Now let's find the best way to pick cards from each partition. First, let's suppose the size of the partition is even, so we have (say) the set {1, 2, 3, 4, 5, 6}. We can only pick either 1 or 2 (but not both), either 3 or 4 (but not both), and either 5 or 6 (but not both), so the maximum number of cards we can pick is 3. But in fact we can achieve this maximum, by picking the cards {1, 3, 5}.

On the other hand, if the size of the partition is odd, so that we have (say) the set {1, 2, 3, 4, 5}, then we can pick either 1 or 2 (but not both), either 3 or 4 (but not both), and possibly 5. So the maximum number of cards we can pick is 3. But we can also achieve this maximum, by picking the cards {1, 3, 5}.

So a suitable algorithm is: find the sizes of the maximal contiguous sequences, divide each by two (adding one if necessary), and add them together to get the maximum possible number of turns.

  int ans = 0;                                // 'ans' will contain the final answer.
  sort(cards.begin(), cards.end());

  int size = 0;                               // 'size' is the size of the current contiguous sequence.
  for(int i = 0; i < cards.size(); i++) 
  {
    int toAdd = 0;                            // If the previous element was the last one in its contiguous
                                              // sequence, we need to process that sequence.
    if(i == 0 || cards[i] == cards[i-1] + 1)
      size++;
    else 
    {
      toAdd = size;                           
      size = 1;
    }

    ans += (toAdd+1)/2;
    if(i == cards.size()-1)                   // The last element of 'cards' must be at the end of a sequence,
      ans += (size+1)/2;                      // so we must process that sequence too.
  }
  return ans;

Here's another solution which also works. We can sort all the numbers in ascending order. Then pick the smallest number remaining in the set (call this number k), and throw it away (as well as any card whose number is k + 1, if it exists). [*] Repeat until there are no more cards left. It is easy to verify that this gives the same choices as the ones described above.

  int ans = 0;
  sort(cards.begin(), cards.end());

  for(int i = 0; i < cards.size(); i++)
    if(i < cards.size()-1 && cards[i] == cards[i+1] - 1) 
    {
      ans++; 
      i++;
    }
    else ans++;

  return ans;

We can prove directly that this greedy algorithm works. Suppose you have an optimal set S of cards which doesn't include the smallest number currently present (call it k). If S is empty, or if the smallest number in S is at least k + 2, just add k to the set and we have a contradiction (since our new set has more cards, and is therefore better than S). So the smallest number in S must be k + 1. Replacing this with k gives an optimal set that contains k. Therefore we can always choose k to lie in our optimal set. This sort of 'exchange argument' is useful for establishing the correctness of many greedy algorithms.

Here are a couple of questions you can use to test your understanding:

  1. In the statement preceding [*], why do we not consider removing cards with the number k - 1?
  2. Suppose you have an undirected rooted tree, and you want to select the maximum number of nodes such that no two selected nodes are joined by an edge. Find a greedy solution for this problem and prove that it works. In what way is the SRMCards problem a special case of this problem?

MafiaGame

Value 500
Submission Rate 84 / 1094 (7.68%)
Success Rate 43 / 84 (51.19%)
High Score travm12 for 397.36 points (15 mins 17 secs)
Average Score 231.37 (for 43 correct submissions)
Used as: Division One - Level One:
Value 250
Submission Rate 484 / 872 (55.50%)
Success Rate 374 / 484 (77.27%)
High Score Petr for 232.83 points (7 mins 49 secs)
Average Score 127.76 (for 374 correct submissions)

This problem was quite a tricky one and required some careful thought. However, once the necessary observations were made, the code needed to implement the solution is fairly short.

We are asked to consider a possible voting system for the party game Mafia. There are N people, and some of them have already nominated a person to lose. The rest then try and 'balance' out the votes as much as possible, by voting for players who currently have the fewest number of votes. The people with the most votes are kept in the pool, and the process is repeated for these people until the pool of losers reduces to one person. That person is the eventual loser.

To get a solution, there are several approaches we could try. One is to use simulation. We can try to keep track, at each stage, of the vulnerable players (initially, all N players are vulnerable). Then, simulate the voting process. If, in any round, the set of vulnerable players doesn't change, then it will never change for all future rounds, and we return 0. Otherwise, the number of vulnerable players must decrease in each round by at least one, so eventually we will reach a stage in which there is exactly one vulnerable player. That player is the losing player.

However, plunging straight into implementing this solution will lead to trouble. For example, if there is a tie, the next voter chooses a candidate 'uniformly at random', so there could be several possibilities. Besides, how do we know that this choice doesn't affect the final answer - i.e., might the number of rounds needed to determine a winner depend on which person a given player votes for? And how do we find the maximum probability that a player will lose? In order to address these issues, we need to observe that one of two possible things must happen in the first round:

  • Each player obtains exactly one vote. This occurs precisely when each of the definite votes is for a different player. Then, the set of vulnerable players does not change, so we return 0.
  • Some player obtains more than one definite vote. Then, after the first round, the vulnerable set consists of exactly those players who started off with the highest number k of definite votes. Why? In the first phase, we need to distribute one vote to each of the players who currently have zero votes. But we won't have enough votes to do this completely. For if we did, each player would have received at least one vote, and some player would have received at least two, which is a contradiction since there are only N votes in total. So, since k ≥ 2, there aren't going to be enough spare votes to raise any player's vote count from k - 1 votes to k votes. Thus, the set of vulnerable players can't change after the definite votes are made.

We know what to do in the first case, so let's focus on the second case. Since the only thing distinguishing different players is the number of definite votes each receives, the probabilities of losing for each of the vulnerable players after the first round must be equal. Thus, we know that if the game terminates in a finite number of turns, the answer is 1/(number of players with the most votes at the start). Otherwise the answer is 0.

So all we need to figure out is whether the game terminates in a finite number of turns. To do this, we can keep track of the number of people in the vulnerable set (since they are all 'the same'). Assume that, after the first round, there are c vulnerable players. Then, they each receive the same number of definite votes, and the other votes are spread evenly. So, after the next round, there are (N mod c) players with (N/c + 1) votes each; the other players will each have N/c votes. Thus, the number of vulnerable players after the next round is (N mod c), or c if (N mod c) = 0.

(For example, suppose that the pre-determined votes are {5, 5, 5}, and there are 7 players who still need to cast their votes. The vote count will then end up being {8, 7, 7} in some order, and this is always going to be the case regardless of the individual random choices made.)

So our algorithm is as follows: maintain the value of c, and keep replacing c by (N mod c) until we either get 0 or 1. In the former case, we return 0, since the vulnerable set remains unchanged. In the latter case, we return 1.0/(number of players with the most votes at the start), since we've reached a state in which exactly one person is vulnerable. This approach is demonstrated in the following code:

  sort(decisions.begin(), decisions.end());

  int currCount = 0;
  int currMaxCount = 0;                         // The current maximum number of votes for a player
  int playersWithMaxCount = 0;                  // The number of players who have this maximum number of votes
	
  for(int i = 0; i < decisions.size(); i++)
  {
    if(i == 0 || decisions[i] == decisions[i-1])
      currCount++;
    else currCount = 1;
	
    if(currCount > currMaxCount)
    {
      playersWithMaxCount = 1;
      currMaxCount = currCount;
    }
    else if(currCount == currMaxCount)
      playersWithMaxCount++;
  }
	
  int c = playersWithMaxCount;
  if(c == decisions.size()) return 0;             // Each player receives one vote
  while(1)
  {
    if(c == 0) return 0;
    if(c == 1) return 1.0/playersWithMaxCount;
    c = N % c;                                    // The loop can't be infinite, since c decreases by at least 1 each time.
  }

GeometricProgressions
Used as: Division Two - Level Three:
Value 1000
Submission Rate 104 / 1094 (9.51%)
Success Rate 4 / 104 (3.85%)
High Score I_L_P for 841.64 points (12 mins 49 secs)
Average Score 625.55 (for 4 correct submissions)

We are given two geometric sequences and asked to find the size of their union. One problem is that the numbers in the sequences could become very large - they could contain up to several hundred thousand digits - so generating these numbers explicitly is impractical. However, each positive integer has a unique prime factorization, and we can use this fact to obtain an efficient solution.

First, however, we examine some special cases. Let's call Si weird if bi = 0, or qi = 0, or qi = 1, or some combination of these. Then, if Si is weird, Si must contain at most two distinct elements (since all subsequent elements will be the same as the second one). So the problem is easy to solve if both sequences are weird.

What if only one sequence (say S1) is weird? Then S2 is an increasing sequence of positive integers, so all its elements are distinct. We can enumerate the elements of S2 one by one until we get to a number that is bigger than 500,000,000. Then all subsequent elements of S2 will be distinct from any element of S1 and any of the previous elements of S2, so we can just add the number of elements remaining in S2 to the total number of distinct integers we've found so far to get the answer. Note that this shortcut saves us from having to do computations with very large numbers.

So the hard case is when neither sequence is weird. We then have two strictly increasing geometric progressions of positive integers, and need to find the size of their union.

We start by trying to represent each possible number more compactly, using its prime factorization. First, we find the possible prime factors that could occur. Since each number in Si is equal to bi times some non-negative power of qi, the set of possible factors is just the union of the prime factors of b1, b2, q1 and q2. There can't be too many such factors - the product of the first ten primes already exceeds 500,000,000 - so there are only at most forty or so. We then decompose each of the numbers b1, b2, q1 and q2 into their prime factorizations. For example, if the set of factors is {2, 3, 7}, we can represent the number 147 as {0, 1, 2}. This means that 147 contains 0 factors of 2, 1 factor of 3, and 2 factors of 7 (i.e., 147 = 20 31 72). We'll call the representation {0, 1, 2} a factorization vector.

Now we can generate each of the sequences explicitly, always representing the numbers in this factorized form. To multiply a number n by qi, it suffices to add the factorization vector of qi to the factorization vector of n. We might need to do this up to around 200,000 times, but this is manageable since there are only at most forty elements per vector. As we generate each element, we store its factorization vector in a set, and finally return the number of elements in that set. The following C++ code illustrates this idea:

  if(b2 == 0 || q2 <= 1)                        // If at least one sequence is weird, make this the first sequence.
    swap(b1,b2), swap(q1,q2), swap(n1,n2);
	
  if(b1 == 0 || q1 <= 1)
  {
    set<long long> S;
    S.insert(b1);
    if(n1 > 1)
      S.insert(b1*q1);                          // Insert the second element of the sequence. All subsequent elements will be the same.
	
    long long curr = b2;
    for(int t = 0; t < n2; t++)
    {
      S.insert(curr); 
      curr *= q2;
      if(curr > 500000000) 
        return (n2-t-1) + S.size();
    }
    return S.size();
  }
  
  set<int> factors;                             // The set 'factors' stores all possible prime factors that can occur.
  factor(b1,factors);                           // factor(i,f) finds the factors of i and inserts them into the set f. This is
  factor(q1,factors);                           //   a standard algorithm which can be implemented using trial division.
  factor(b2,factors);
  factor(q2,factors);

  vector<int> F(factors.begin(), factors.end());
  vector<int> repb1, repq1, repb2, repq2;       // The prefix 'rep' stands for 'representation'.
  decompose(b1,repb1,F);                        // decompose(i,v,F) finds the factorization vector v of i, based on the factors in F.
  decompose(q1,repq1,F);
  decompose(b2,repb2,F);	
  decompose(q2,repq2,F);	

  set<vector<int> > S;
  	
  for(int i = 0; i < n1; i++)
  {
    S.insert(repb1);
    for(int t = 0; t < repb1.size(); t++)
      repb1[t] += repq1[t];
  }
  for(int i = 0; i < n2; i++)
  {
    S.insert(repb2);
    for(int t = 0; t < repb2.size(); t++)
      repb2[t] += repq2[t];
  }
  return S.size();

Here are some questions for investigation that you might find interesting:

  1. Can you think of a way to handle weird sequences in the main part of the code, so that the preliminary case analysis is not needed? (Hint: introduce a special factorization vector to represent the integer 0.)
  2. Write pseudocode for the factor and decompose routines. (For factor(i,f), note that if you can't find a factor less than or equal to the square root of i, then i must be prime. Since i is at most 500,000,000, you don't need to consider factors that are greater than about 10,000*sqrt(5). Thus, the routine will run quite quickly.)
  3. In the above code, the factorization vectors are represented as vector<int>s, rather than vector<long long>s, to save space. However, is it possible to find an input that could make the vector<int>s overflow? Why, or why not?
  4. What is the maximum possible size of the set of factors, over all possible valid inputs? (40 is quite a loose upper bound.)
  5. In the actual contest, the four accepted solutions for this problem worked slightly differently. Instead of using the prime factorization of an integer to represent it, they represented the integer by its value modulo two or more numbers. (Usually, the numbers chosen were primes, but in one case, the number 123,155 was used.) Although this approach is likely to give the correct answer in most cases, it isn't clear to me why it is guaranteed to work for all inputs - the danger here is that distinct elements may have the same representation. So, in order for this to be a correct solution, we need to show that this kind of 'hash collision' cannot take place. Each of the four accepted solutions uses a slightly different algorithm. For each solution, can you supply either a proof that it works, or a successful challenge case?

FractalPicture
Used as: Division One - Level Two:
Value 500
Submission Rate 87 / 872 (9.98%)
Success Rate 41 / 87 (47.13%)
High Score ainu7 for 370.65 points (18 mins 10 secs)
Average Score 257.06 (for 41 correct submissions)

First, let's consider the final segments. The process described is recursive: we start with an active root at (0,0), and draw a line from (0,0) to (0,54). Then we get a new root at (0,54), which extends in three different directions. For each of these directions, we consider the rectangle bounding successive generations of the fractal in that direction, and take its intersection with the original rectangle. Then we rotate and scale the diagram so that (0,0) corresponds to the location of the new root, and the direction of the positive y-axis corresponds to the direction in which the next line will be drawn. This gives us another instance of the same problem, but with one fewer generation.

To take a concrete example, suppose the original rectangle had its corners at (-10,54) and (10,55). The first line, from (0,0) to (0,54), does not intersect this rectangle in a line, so it doesn't contribute anything to the answer. The new root is at (0,54), and the rectangles covered by successive generations of each of its three branches are (-27,45)-(0,63), (-9,54)-(9,81), and (0,45)-(27,63). The intersections of the original rectangle with each of these domains are (-10,54)-(0,55), (-9,54)-(9,55) and (0,54)-(10,55) respectively. [*] These intersections need to be scaled by 3 and rotated to obtain new copies of the original problem, on which we repeat the process.

We therefore obtain the following pseudocode:

  Solve(Generation g, Rectangle r)
    Ans := 0
    Draw the line from (0,0) to (0,54) and add the length of its intersection with r to Ans
    Ans += Solve(g+1, scaled and rotated intersection of r with the rectangle (-27, 45)-( 0, 63))/3
    Ans += Solve(g+1,             scaled intersection of r with the rectangle ( -9, 54)-( 9, 81))/3
    Ans += Solve(g+1, scaled and rotated intersection of r with the rectangle (  0, 45)-(27, 63))/3
    Return Ans

This calculation branches three times on every call of Solve(), so it will time out. However, we can save a huge amount of work by adding a couple of checks at the beginning. Firstly, if the rectangle passed to the function is empty, we can immediately return 0. Also, if r contains the entire rectangle (-27,0) to (27,81), then all successive generations of the corresponding part of the fractal will also be contained in r, and we can calculate the total lengths for all these generations in constant time. This is done as follows: note that, in the original fractal, we draw a line of length 54 in generation 2, three lines of length 18 each in generation 3, nine lines of length 6 each in generation 4, and so on. In other words, we always add a total length of 54 in each generation. So the total length added for the whole fractal is simply 54 times the number of generations left.

A similar analysis can be done for the non-final segments. Notice that in the second generation we draw a final segment of length 54, and the total length of the non-final segments is 27*3 = 81. In the third generation, the total length of the non-final segments is 9*9 = 81. In fact, the total length of the non-final segments is always 81. So once we get to a stage where the rectangle encloses (-27, 0) to (27, 81) entirely, we can just return the sum of the remaining lengths within this box, which is (number of remaining generations)*54 + 81.

It is important to note that all the coordinates of the original rectangle are integers - so, in five generations, we'll reach a stage in which r contains the entire rectangle. After that, there's not going to be very much branching, so the code will run in time. (There is an exception to this: if our rectangle is (-27,0)-(0,81) or (0,0)-(27,81), the intersection of this rectangle with the top branch of the fractal will result in the same rectangle. We could hard-code the answer for this case explicitly, but this is not necessary - since there are only 500 generations and no significant branching, the code still runs very quickly.)

Here is some code illustrating this idea. The answer to the problem can be obtained by calling solve(1, x1, y1, x2, y2).

  solve(int gen, int x1, int y1, int x2, int y2)		// Calculate total length of segments in generation 'gen' onwards
    if(x1 >= x2 || y1 >= y2) return 0;                          // Not a valid rectangle
    if(gen == 501) return 0;
	
    if(x1 <= -27 && y1 <= 0 && x2 >= 27 && y2 >= 81)
      return (500-gen)*54 + 81;
	
    double ans = 0;
    if(x1 <= 0 && x2 >= 0)
      ans += max(min(y2,54) - max(y1,0), 0);                    // Add the contribution of the line from (0,0) to (0,54)
	
    ans += solve(gen+1, max(3*(y1-54) ,-27), max(-3*x2    ,0), min(3*(y2-54) ,27), min(-3*x1    ,81))/3;  // left branch
    ans += solve(gen+1, max(3*x1      ,-27), max(3*(y1-54),0), min(3*x2      ,27), min(3*(y2-54),81))/3;  // top branch
    ans += solve(gen+1, max(3*(-y2+54),-27), max(3*x1     ,0), min(3*(-y1+54),27), min(3*x2     ,81))/3;  // right branch
	
    return ans;

We are given the sum and the product of the digits of a number N, and need to find how many such N have this sum and product.

Firstly, note that no digit in N can be 0. Otherwise, the product of its digits will be zero, which is not allowed by the constraints. Now let di be the number of 'i' digits in N. Then

   S = d1 + 2*d2 + ... + 9*d9 .

Now consider p2. Each '2' in N contributes 1 to p2, each '4' contributes 2, each '6' contributes 1, and each '8' contributes 3. All other digits contribute nothing. Therefore

   p2 = d2 + 2*d4 + d6 + 3*d8.

Similarly, 

   p3 = d3 + d6 + 2*d9,
   p5 = d5,
   p7 = d7,

and these are all the constraints we have.

Suppose we fix d6. Then we can write

 d1 + 2*d2 + 3*d3 + 4*d4 + 8*d8 + 9*d9  =  S - (5*p5 + 6*d6 + 7*p7)
 d2 + 2*d4 + 3*d8                       =  p2 - d6
 d3 + 2*d9                              =  p3 - d6

Now let's fix d8, d9 and d4. Note that d8 can be bounded above by (p2 - d6)/3, d4 can be bounded above by (p2 - d6 - 3d8)/2, and d9 can be bounded above by (p3 - d6)/2. So we have to examine at most 100 * 33 * 50 * 50 ~ 8000000 choices of the parameters d6, d8, d9 and d4 (and this number is actually quite an overestimate). Once we've fixed d6, d8, d9 and d4, we can find the other parameters as follows: 

 d2 =  p2 - d6 - 2*d4 - 3*d8
 d3 =  p3 - d6 - 2*d9
 d1 =  S - (5*p5 + 6*d6 + 7*p7) - (2*d2 + 3*d3 + 4*d4 + 8*d8 + 9*d9).

If any of the di's are negative, this choice is invalid and we skip it. Otherwise, all we need to do is to find the sum of the numbers which have d1 '1's, d2 '2's, ..., and d9 '9's. [*] The remainder of the editorial explains how to do this.

First, let's introduce some variables that will come in handy. Let L be equal to the length of the number, so L = d1 + d2 + ... + d9. Also, let K denote how many such numbers there are satisfying the criterion [*] in the previous paragraph. Then K is simply L!/(d1!d2!...d9!).

In order to figure out how to find the sum of the numbers satisfying [*], let's try a small example. Suppose we want to find the sum of all numbers with two '1's, one '2' and one '4' (so L = 4). Then K = 4!/(2!1!1!) = 12, and these twelve numbers are:

  1124
  1142
  1214
  1241
  1412
  1421
  2114
  2141
  2411
  4112
  4121
  4211

Notice that in the first column, there are six 1's, three 2's and three 4's. This is actually the case for each of the four columns! And this is clear in retrospect: since every possible permutation is considered, half of the digits in each column should be 1's, a quarter should be 2's, and a quarter should be 4's. So the sum of the digits in each column is the same. This means that, in order to find the overall sum, we just need to find the sum of the digits in one column, and multiply this by m = 11..1, where the number of 1's in m is equal to L.

In this example, there are (d1/L)K '1's, (d2/L)K '2's, and (d4/L)K '4's in the first column. So our answer is 

  ((d1*1 + d2*2 + d4*4) * K / L) * m.

Now for the finicky details. Notice that we have to divide by L, and we have to do this modulo 500,500,573. But 500,500,573 is prime, so this means we can apply Fermat's little theorem to find the inverse L-1 of L, and then we can multiply the numerator above by L-1. Fermat's little theorem says that we can find L-1 by computing L500,500,571 modulo 500,500,573. This can be done quickly by repeated squaring.

However, performing repeated squaring each time we want to compute an inverse is suboptimal, since L can take only about 2500 different values. (Why?) So we might as well precompute these inverses at the very beginning, rather than compute them from scratch each time we need them. Similarly, for the calculation of K, we can precompute the factorials and their inverses instead of computing them from scratch each time. We can also precompute the values of 11..1 mod 500,500,573 for different values of L. This allows us to compute the answer for a given set of di very quickly.

In short, the algorithm is as follows: try all possible sets of di, compute the answer for each set, then add the answers up and return the result. You can find a clean implementation of these ideas in neal_wu's code. The choice of variables he chooses to brute-force over are slightly different, but the underlying ideas are similar to those described above.
