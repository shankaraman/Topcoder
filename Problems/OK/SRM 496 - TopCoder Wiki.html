Single Round Match 496
Tuesday, February 1st, 2011

Match summary

The first SRM of February drew 1640 coders, and was somewhat easier than the previous three matches - in particular, 19 people solved the Div 1 Hard problem. (For comparison, in the three January 2011 matches, only 1, 2 and 5 coders successfully solved the Div 1 Hard.)

The problems in both divisions were nicely balanced, and none were too implementation-heavy or required lots of optimization. Usually, the hardest part was getting the right idea in the first place.

In Division 1, fourteen contestants solved all three problems. The winner was ACRush, who solved all the problems in 32.5 minutes and, on top of that, scored 4 successful challenges. He was followed by pieguy (who solved everything in 36 minutes), and UdH-WiNGeR (who took 37.5 minutes). At this rate, each of them should be able to complete two SRMs in a single time slot...

Division 2 was won by three coders from China, including two newcomers to TopCoder. These three coders were the only ones to solve all three problems, and they consequently won the top three places by a significant margin. In first place was cypress_wx (who, again, used less than half the SRM time to solve all three problems!), followed closely by Sevenkplus. 1010 rounded off the top three.

The Problems

AnagramFree
Used as: Division Two - Level One:
Value 250
Submission Rate 772 / 949 (81.35%)
Success Rate 692 / 772 (89.64%)
High Score scarllet for 249.61 points (1 mins 7 secs)
Average Score 191.58 (for 692 correct submissions)

We are given a set of strings S, and need to find the size of the largest subset A of S such that no two strings in A are anagrams of each other.

First, let's figure out when two strings a and b are anagrams of each other. This happens precisely when the letters used to build a and b are the same. So, for each string in S, we just need to store the set of letters needed to build it, and count how many distinct such sets there are.

A simple way to do this is to sort the contents of each string a in S, since two strings which are anagrams would then be sorted to the same string. For example, the strings "abgska" and "kaagsb" would both be transformed to "aabgks", and so would any other anagram of "abgska".

All that is left is to find how many distinct sorted strings there are. This can be done by, for example, putting all the sorted strings into a set data structure (which is available in most programming languages), and returning the size of the set.

Here is an example of a short C++ implementation:

   set<string> A;                      // Declare a set A, which will contain the sorted strings
   for(int i = 0; i < S.size(); i++)
   {
      sort(S[i].begin(), S[i].end());  // Convert each string into its sorted form...
      A.insert(S[i]);                  // ... and put it into A
   }
   return A.size();                    // Return the number of elements of A

Let's end with a little extension. Given a set of strings S, can you design an algorithm that calculates the minimum size of a subset of S that is not anagram-free (or returns -1 if such a set does not exist)?

ColoredStrokes
Used as: Division Two - Level Two:
Value 500
Submission Rate 564 / 949 (59.43%)
Success Rate 390 / 564 (69.15%)
High Score Marchenko.Igor for 492.24 points (3 mins 34 secs)
Average Score 336.60 (for 390 correct submissions)
Used as: Division One - Level One:
Value 250
Submission Rate 683 / 691 (98.84%)
Success Rate 603 / 683 (88.29%)
High Score tourist for 247.38 points (2 mins 56 secs)
Average Score 210.95 (for 603 correct submissions)

In this problem, we need to find the minimum number of horizontal red strokes and vertical blue strokes needed to obtain a given picture.

Let's think about the red strokes first. Any square marked 'R' or 'G' is part of a red stroke, and all other squares are not. So let's consider a new bitmap in which the 'R' and 'G' cells are replaced by 'R' cells, and all other cells are replaced by '.' cells. Then the problem reduces to the following: given a bitmap consisting of '.' and 'R' only, what is the minimum number of horizontal strokes needed to paint all the 'R' cells and none of the '.' cells?

It is clear that we can consider each row independently, since each horizontal stroke only affects one row. So let's suppose we have a row of the form

   ...R...R.RR.RRRR

What is the fewest number of horizontal strokes needed to achieve this coloring? Clearly, it is the number of maximally connected regions consisting only of 'R' cells, since we can paint each such region with a single stroke. In the example, there are four such regions, consisting of 1, 1, 2 and 4 cells respectively. We can thus determine the answer by traversing the row from left to right, keeping track of whether we are currently in a '.' region or an 'R' region, and incrementing a counter each time we enter a new 'R' region.

Counting the blue strokes is no harder. We can do almost exactly the same thing, except that the bitmap now needs to be divided into vertical stripes and we need to look at blue cells instead of red ones. Then, we can sum the contributions from both colors, and return the result as our answer.

Here is some code that illustrates this procedure:

   int ans = 0;
   for(int i = 0; i < picture.size(); i++)
   {
      int inStroke = 0;                                  // inStroke is 0 if we aren't in the middle of a stroke, and 1 otherwise
      for(int j = 0; j < picture[i].size(); j++)
      if(picture[i][j] == '.' || picture[i][j] == 'B')   // If the cell is not red...
         inStroke = 0;                                   // ... we are not in the middle of a stroke
      else if(inStroke == 0)                             // Otherwise, if the cell is red and we weren't in a stroke before...
         inStroke = 1, ans++;                            // ... we've just entered a stroke, and must increment ans
   }
	
   for(int j = 0; j < picture[0].size())                 // Do exactly the same thing for vertical stripes
   {
      int inStroke = 0;
      for(int i = 0; i < picture.size(); i++)
      if(picture[i][j] == '.' || picture[i][j] == 'R')
         inStroke = 0;
      else if(inStroke == 0)
         inStroke = 1, ans++;
   }
   return ans;

Finally, a quick question for the reader. Notice that the problem statement contains the phrase "Manao never draws two horizontal or two vertical strokes that overlap". How would you solve the problem if this restriction was removed?

PalindromfulString
Used as: Division Two - Level Three:
Value 1000
Submission Rate 14 / 949 (1.48%)
Success Rate 3 / 14 (21.43%)
High Score cypress_wx for 640.13 points (18 mins 13 secs)
Average Score 563.81 (for 3 correct submissions)

We need to count the number of strings of length N (where N is at most 11) which have at least K substrings of length M that are palindromes.

A first approach might be to try and enumerate all possible strings. But this takes too long - there can be up to 2611 = 3 × 1015 such strings.

However, notice that for the purposes of this problem, strings like ABBAB, BAABA and CAACA are equivalent. If one of them contains exactly p palindromes of length M, the others do too. This is because the condition that a string is a palindrome is a condition that certain pairs of characters of the string are equal to each other. We do not care what these characters are! Therefore the condition that a string contains exactly p palindromes of length M is also a condition that certain pairs of characters of the string are equal or not equal to each other. The actual characters used are not important.

So let's try and enumerate all possible 'letter patterns' of length N, which seems much more reasonable - there can't be that many. We can find the letter pattern of a string as follows: let c0, c1, c2, ... be the distinct characters in the string, ordered by the first time they are encountered when we read the string from left to right. Then replace all c0's by 'a's, all c1's by 'b's, and so on. So, for example, if we had the string "AJNFHALJJFJ", the characters we encounter would be A, J, N, F, H, and L. We replace these by a, b, c, d, e and f respectively, to get the letter pattern "abcdeafbbdb". Notice that every string of length N can be converted to a letter pattern, and that this conversion is unique.

A recursive algorithm can then be used to generate all possible letter patterns and store them. For example, we can call the function below with parameters N, "" (the empty string) and 'a':

   // In this code, we assume that the variables 'M', 'K' and 'ans' are defined globally.

   void construct(int rem, string soFar, char nextChar)  // Suppose you've constructed a pattern 'soFar', and used characters 
   {                                                     //   'a' to nextChar-1, and you have 'rem' more characters to add.
      if(rem == 0) 
      {                                                  // If 'rem' = 0, we've built a letter pattern! 
         int tot = 0;                                    // We now need to check whether it is palindromful.
         for(int t = 0; t < soFar.size()-M+1; t++)
         {
            int first = t;
            int last = t+M-1;

            int bad = 0;                                 // This tells us whether a particular substring is a palindrome.
            while(last >= t)
            {
               if(soFar[first] != soFar[last])
               {
                  bad = 1;
                  break;
               }
               first++, last--;
            }
            if(bad == 0) tot++;
         }
				
         if(tot >= K)                                    // If tot >= K, the pattern is palindromful.
            ans += calcAns(nextChar);                    // calcAns() will determine the number of patterns that map to this 
         return;                                         //   letter pattern (see below).
      }

      construct(rem-1, soFar+nextChar, nextChar+1);      // Try adding character 'nextChar' to the end of the pattern
      for(char c = 'a'; c < nextChar; c++)               // Try adding each character used so far to the end of the pattern
	 construct(rem-1,soFar+c,nextChar);
      return;
   }

It turns out that, in the worst case (N = 11) there are only 678570 letter patterns, so our assumption that there aren't too many letter patterns is a good one. (We can check this locally quite easily by adding a counter to the above code. An alternative way of obtaining this number is hinted at in the last paragraph, below.)

Now, given a letter pattern, we can easily check whether the words it represents contain at least K substrings of length M that are palindromes, since this is true for those words if and only if it is true for the letter pattern itself. So, given a valid letter pattern, we just need to figure out how many strings it corresponds to. How can we do this?

Let's look at an example. Consider the letter pattern "abcdeafbbdb". The 'a' could represent any of 26 characters. Once the character it represents has been determined, the 'b' could represent any of the remaining 25 characters, and so on. So, in this case, there are 26 × 25 × 24 × 23 × 22 × 21 words that map to this letter pattern. In general, if a letter pattern contains k different characters, it represents 26 × 25 × ... × (26-k+1) words. This calculation is illustrated in the code snippet below: 

   long long calcAns(char nextChar)
   {
      long long summand = 1;
      long long base = 26;
      for(int p = 0; p < nextChar-'a'; p++)
         summand *= base, base--;	
      return summand;
   }

Finally, we just need to add the contributions from all possible letter patterns, and return the answer.

If you would like to brush up on your dynamic programming skills, here's a related exercise that you can try. Let Bn be the number of letter patterns of length n, and suppose you've already calculated B0, ..., Bn-1. Can you find a recurrence relation that can be used to calculate Bn? (If you're stuck, look up 'Bell numbers' on Google.)

OneDimensionalBalls
Used as: Division One - Level Two:
Value 500
Submission Rate 327 / 691 (47.32%)
Success Rate 186 / 327 (56.88%)
High Score ACRush for 428.90 points (11 mins 59 secs)
Average Score 254.26 (for 186 correct submissions)

We are given two pictures of balls, and need to match balls in the first picture to those in the second. One very useful piece of information is that all the balls move at the same speed for the same amount of time. So, between the two pictures, each ball moves the same distance D, either left or right.

How do we find the possible values for D? We know that every ball in the first picture can be found somewhere in the second picture, so let's try matching the leftmost ball in the first picture (call this ball b) to each of the balls in the second picture in turn. Taking the absolute value of the difference between their positions gives a possible value of D.

(There is one small thing to be careful about here. Since the velocities are all positive, and a positive amount of time elapses between the taking of the two pictures, we should not consider cases in which D = 0.)

So now let's assume we've found the ball in the second picture that is matched to b (which, conveniently, also fixes D). We can get rid of b and its partner, and then construct a bipartite graph, joining two balls in different pictures by an edge if and only if they are separated by a distance D. We then need to count the number of maximum matchings in this graph, and then we're done. Unfortunately, this is a #P-complete problem, and if we can solve it in polynomial time, we would have proved that P = NP. That would make this problem quite a bit trickier than the usual Division 1 Medium.

But notice that the graph we obtain in this way is actually rather special, since each node in the graph has at most two neighbors! (The two possible edges ending at a node a correspond to matching a with a ball in the other picture that is located a distance D to the left, or a distance D to the right, of a.) In fact, this just means that the graph consists of a set of disjoint paths, whose nodes alternate between the two pictures.

(Actually, in general, a graph whose nodes have degree at most 2 would consist of paths and cycles. However, it's impossible for cycles to occur in our graph. Can you see why?)

For each path, we then need to figure out the number of matchings that include all balls from the first picture. Depending on where the paths start and end, there are four cases:

(a) The path starts and ends with balls in the first picture. In this case, it is impossible to find a matching containing all the balls in the first picture, since we don't have enough balls from the second picture to use.

(b) The path starts with a ball in the first picture, and ends with a ball in the second picture. Then there is exactly one possible matching.

(c) The path starts with a ball in the second picture, and ends with a ball in the first picture. This is similar to case (b), and there is exactly one possible matching.

(d) The path starts with a ball in the second picture, and ends with a ball in the second picture. So there are (say) k balls in the first picture, and k + 1 balls in the second. This allows k + 1 possible matchings. (In each matching, all balls in the first picture are connected to their left neighbour up to a certain point, and thereafter all remaining balls are connected to their right neighbour.)

Finally, since each component is independent of the others, we can get the total number of matchings by multiplying the answers found for each component. We can repeat this for each matching of b to a ball in the second photograph, and then sum the results to get our final answer.

Here is some code that illustrates these ideas. (For convenience, let fP denote the firstPicture array, and sP denote the secondPicture array.)

   sort(fP.begin(), fP.end());

   long long ans = 0;

   for(int j = 0; j < sP.size(); j++)
   if(fP[0] != sP[j])
   {
      long long D = abs(sP[j]-fP[0]);                     // Match fP[0] with sP[j]
		
      set<long long> S(sP.begin(),sP.end());              // S is a set storing the balls from sP we haven't used yet
      S.erase(sP[j]);                                     // Remove the ball at sP[j]; it's already been matched		

      vector<long long> F(fP.begin()+1, fP.end());        // Remove the ball at fP[0]; it's already been matched
		
      long long a = 1;                                    // This stores the partial answer for this choice of fP[0]'s partner

      while(F.size() > 0)
      {
         long long curr = F[0];                           // Find the component of the leftmost remaining ball ('curr') from
         F.erase(F.begin());                              //   the first picture
			
         int up = 1;                                      // 'up' = number of balls from fP that are used in this component
         int down = 0;                                    // 'down' = number of balls from sP that are used in this component
         if(S.find(curr-D) != S.end())                    // Check if there is a ball in S that is D to the left of 'curr'
            S.erase(curr-D), down++;

         while(S.find(curr+D) != S.end())                 // Try and find a ball in S that is D to the right of 'curr'
         {
            S.erase(curr+D), curr += D*2, down++;       
            int ok = 0;
            for(int p = 0; p < F.size(); p++)             // Look for the new 'curr' in F (which should be 2*D 
            if(F[p] == curr)                              //   to the right of the old one)
            {
               ok = 1;
               F.erase(F.begin() + p);
               up++;
               break;
            }
            if(ok == 0) break;                            // If there is no such new 'curr', the component ends here
         }
	
         if(up+1 == down)                                 // This is case (d) above
            a *= down;
         else if(up == down+1)                            // This is case (a) above
         {
            a = 0;
            break;
         }
      }
      ans += a;                                           // Otherwise we are in case (b) or (c)
   }
   return ans;

YetAnotherHamiltonianPath
Used as: Division One - Level Three:
Value 950
Submission Rate 97 / 691 (14.04%)
Success Rate 19 / 97 (19.59%)
High Score pieguy for 791.76 points (13 mins 15 secs)
Average Score 540.49 (for 19 correct submissions)

In this problem, we are given a weighted graph and need to find the cheapest Hamiltonian path in it. The start and end nodes of the desired path are given. In general, this problem is NP-hard, so there is probably something about the graph and its weights that allows us to compute the answer quickly.

Let's try and make the problem simpler. A Hamiltonian path in this problem corresponds to a permutation of the nodes, starting at node 0 and ending at node 1. The cost of the path is the sum of Li2 + Li+12 - LCP(Li, Li+1)2 for all consecutive pairs (i, i+1) in this list, where Li is the length of the label of node i and LCP(i, j) is the length of the longest common prefix of i and j. We can see that, regardless of the ordering of the nodes, the first two terms in each summand count each label twice except for the first and last labels. So we can define a base cost as follows:

  baseCost = L02 + L12 + 2(L22 + L32 + ...)

and then the problem reduces to finding the permutation that maximizes the sum of the LCP(Li, Li+1)2 for all consecutive pairs (i, i+1).

It seems easier to come up with a correct solution for the next part of the problem than to find a proof that it works! In this editorial I will describe a very elegant solution by UdH-WiNGeR, and its proof; later on I will outline other solutions that also work.

We start by modifying the problem slightly. Instead of finding the most expensive Hamiltonian path from node 0 to node 1, let's find the most expensive Hamiltonian cycle such that node 0 and node 1 are adjacent in the cycle. Given the cost of this cycle, we can then subtract LCP(L0, L1)2 to get the cost of the most expensive Hamiltonian path from node 0 to node 1.

In fact, let's consider the more general problem of finding the most expensive Hamiltonian cycle in the graph (without any restriction on nodes 0 and 1). We'll then show that this cycle can be converted into one with nodes 0 and 1 adjacent, without changing its cost.

We'll try and use the special structure of the weights to construct a solution. First, observe that if we fix a non-negative integer N, then we can merge all nodes whose labels have a common prefix of length at least N into a single set. For example, if N = 2 and the labels are "aaba", abaa", "abac" and "ab", we can form two sets {"aaba"} and {"abaa", "abac", "ab"}. Now note that any edge between nodes in the same set costs at least N2, and any edge between nodes in different sets costs at most (N-1)2. So we'd like to use more 'within-set' edges than 'between-set' edges.

Suppose that, for a given N, we have TN > 1 sets. Using the above observation, we can see that, in any maximum-cost Hamiltonian cycle, we require exactly TN edges to join nodes in different sets together, and the rest of the edges to join different nodes in the same set. (We need at least TN 'between-set' edges in order to visit all the sets. We should not exceed TN because the cost of using a 'between-sets' edge is strictly smaller than the cost of a 'within-set' edge.)

Hence, ideally, we'd like to construct a cyclic ordering of the labels such that, for any N, all labels with a common prefix of length at least N occur consecutively in the cycle. This will guarantee that, for each N, exactly TN 'between-set' edges are used.

One way to do this is to sort the labels lexicographically. For such an ordering, it is easy to check that, given any prefix P, all labels with that prefix form a consecutive block of elements in the cycle. The cost of this cycle is therefore the cost of the maximum Hamiltonian cycle.

But wait! In this cycle, nodes 0 and 1 may not be adjacent. Can we fix this?

Fortunately, we can. To see this, think about the prefix tree (or trie) constructed using the labels of the nodes. The trie can be constructed so that the labels of its leaves, taken from left to right, correspond to our lexicographical ordering. This is because all strings with common prefixes occur consecutively in our list. (A technical note: in order to ensure that every label corresponds to a leaf node, we should add a special delimiter character to the end of each label, and this delimiter should be different for each label. This is to ensure that no label is a prefix of another label, since in that case the former label would correspond to an internal node of the trie. Note that this doesn't change the LCPs, and if the delimiter characters are chosen to be smaller than the letters used in the labels ('a'-'z'), the lexicographical ordering won't be affected either, so this small modification doesn't change our overall algorithm.)

The Hamiltonian cycle is then represented by the labels of the leaves, read in order from left to right. Now notice that, in the trie, we can permute the direct children of each node arbitrarily, and this does not change the cost of the cycle along the leaves. Using a sequence of such permutations, we can make the start and end leaves neighbors. This gives a Hamiltonian cycle in which nodes 0 and 1 are adjacent, and leaves the cost of the cycle unchanged.

And that's it! In summary, we sort all the labels, calculate the cost of the Hamiltonian cycle thus formed, and subtract LCP(L0, L1)2. Finally, subtract this cost from the base cost, and return the answer. Easy to code, but tricky to prove. (In the following code snippet, we assume that the function LCP() is already implemented.)

   int baseCost = 0;
   for(int i = 0; i < label.size(); i++)
   if(i < 2)
      baseCost += label[i].size() * label[i].size();
   else
      baseCost += 2 * label[i].size() * label[i].size();
	
   string start = label[0];
   string end = label[1];

   sort(label.begin(), label.end());
   label.push_back(label[0]);

   int maxCost = 0;                                                     // This is the maximum cost of a Hamiltonian cycle
   for(int i = 0; i < label.size()-1; i++)
      maxCost += LCP(label[i], label[i+1]) * LCP(label[i], label[i+1]);
   maxCost -= LCP(start, end) * LCP(start, end);                        // Remove the edge joining 'start' and 'end'

   return baseCost-maxCost;

Here are some other solutions for the second half of this problem. You might like to try and find proofs for them, perhaps using some of the observations made above.

1. Greedily grow the path from both ends. That is, at each step, find the label s among those unused so far that maximizes max(LCP(s, start), LCP(s, end)). (Initially, 'start' = L0 and 'end' = L1.) If the maximum is found among LCP(s, start), then replace start with s, add the cost of (s, start) to a running total, and remove s from the remaining labels. Similarly for LCP(s, end). Repeat until there are no labels left.

2. Begin by sorting all edges in order of decreasing weight. Then iterate along this sorted list and add an edge to the path if it doesn't invalidate it (e.g., create a cycle, or increase the degree of a vertex to 3, etc.).

(As an aside, note the resemblance of these two algorithms to the MST algorithms of Prim and Kruskal.)

3. Start with N = 0 (using the notation above), so that all nodes are in the same set. Then increment N. This may break some sets into smaller ones. Each time this occurs, we must join the smaller sets with edges of length (N-1)2 and add the weights of these new edges to the overall cost. (We may have to use TN edges or TN - 1 edges, depending on whether nodes 0 and 1 are in the same set or not.) Continue incrementing N until all the sets have been broken into sets of one node each; then we are done, since this means we've used label.size()-1 edges. Note that, for this approach, it is not necessary to construct the path explicitly; maintaining an overall running cost is sufficient.
