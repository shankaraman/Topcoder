Single Round Match 516
Tuesday, August 30th, 2011

Match summary

SRM 516 started fast when many division 1 coders were seemingly able to solve at least two problems. The challenge phase and subsequent system test phase proved to be full of casualties and at the end less than 100 coders had more than one correct submission. The top performers had concerns of their own, the hard problem was solved by only 6 coders which needed speed and great challenging skills to reach the very top. With his outstanding speed and 150 challenge points, Petr scored yet another SRM win. It was not easy as tourist was only 10 points bellow the first place. The close race does not end there, wata and rng_58 acquired the third and fourth place respectively and were still less than 50 points bellow Petr.

In division 2, the need for speed was as strong as ever, with almost 30 coders solving the hard problem and 11 solving all three of them. rajatag12 returns to division 1 after a division 2 win with great scores in the problems and 200 challenge points. Former gray coder Try_to_try got the second place thanks to the fastest 1000 submission. Third place went to safariworld who joins division 1 after his 25-th match.

The Problems

NetworkXZeroOne
Used as: Division Two - Level One:
Value 250
Submission Rate 831 / 925 (89.84%)
Success Rate 643 / 831 (77.38%)
High Score liuzaoxin for 249.99 points (0 mins 12 secs)
Average Score 197.41 (for 643 correct submissions)

The key observation was to notice the pattern that determines whether a string of x and o characters follows the required rule or not. It can be inferred from the examples and it can be confirmed by a quick analysis.

Let us focus on the simplest case: making sure that every length-2 substring contains an equal number of x and o characters. Since the length is 2, the only way for such substring to contain equal amounts of x and o characters is to be xo or ox. Note that every length-2 substring must follow this property. If we wanted a string of 5 characters, and we decided the first pair of characters was xo, then we would forcibly have ox for the second pair of characters (the letter o has already been decided). Then we can get the remaining characters: xoxox. The only other possibility for a string of length five that follows the property for length 2 substrings is oxoxo. By the same logic, we can conclude that in order to fulfill the length-2 property, we need sequences of alternating characters: oxoxox.... or xoxox... .

Let us now consider substrings of length 4. This time we have to place 2 x and 2 o characters. The string is already either of the form oxoxo... or xoxox... because the rule for length-2 substrings must also be fulfilled. Note that oxox and xoxo fulfill the rule for length-4 substrings as well.

For every even-length substring, if we assume that the substring follows an alernating pattern xoxox... or oxoxox... , the rule for the new even length will also be fulfilled. Since those alternating patterns are necessary for the length-2 substrings property and sufficient for the properties of the other even length substrings, we can take the conclusion that there are only two different patterns that follow the interesting property: xoxoxo... and oxoxoxo...

Building the two strings that match the pattern and have the same length as the input message is simple. Once they are built we can check if the message is compatible with that pattern. All characters different to ? in the message must be equal to the equivalent position in the built pattern of alternating characters. Once the alternating pattern that is compatible with the message is found, return it. The statement guarantees that exactly one of the patterns will be compatible with the message.

string reconstruct(string message)
{
    // The s variable decides whether the
    // alternating pattern 
    // begins with x or o
    for (int s=0; s<2; s++) {
        string fixed = message;
        
        // ch holds the current character,
        // it alternates between x and o for each position
        char ch = (s? 'o' : 'x' );

        bool bad = false;
        for (int i=0; i<message.size(); i++) {
            if ( message[i] == '?' ) {
                fixed[i] = ch;
            } else if ( message[i] != ch ) {
                // The non-? characters in the message must coincide
                // with those in the pattern we are building.
                bad = true;
            }

            ch = ( (ch=='x') ? 'o' : 'x');
        }
        // We found the pattern that matches the message:
        if (! bad ) {
            return fixed;
        }
    }
    return "!";
}

NetworkXOneTimePad
Used as: Division Two - Level Two:
Value 500
Submission Rate 657 / 925 (71.03%)
Success Rate 212 / 657 (32.27%)
High Score mathscholar for 484.84 points (5 mins 3 secs)
Average Score 292.29 (for 212 correct submissions)
Used as: Division One - Level One:
Value 250
Submission Rate 686 / 700 (98.00%)
Success Rate 461 / 686 (67.20%)
High Score embe for 246.47 points (3 mins 24 secs)
Average Score 190.91 (for 461 correct submissions)

Let us first begin with a small variation: we have only one plaintext and one ciphertext. Is it possible to build the key? Due to how the XOR operation works, the solution is as simple as: (plaintext XOR ciphertext). (xor is associative: we have (a xor (a xor b)) = (a xor a) xor b = b.)

In the problem there are multiple plaintexts and ciphertexts. Note that, given a key, each ciphertext must have one plaintext as its original message before using the key. The previous paragraph is handy in that there is a finite number of potential keys. Take ciphertexts[0], one of the plaintexts must be the original message for ciphertexts[0]. We can just iterate through each plaintexts[i], find the key that turns plaintexts[i] into ciphertexts[0] and verify if that key also works for the remaining ciphertexts. Since there are no repeated values among plaintexts, we will only check each potential key once.

Verifying if a key is valid for the entire input is not difficult. We can either change each plaintext into new ciphertexts and then verify that all of the input ciphertexts form a subset of the set of newly-found ciphertexts. Or we can go the reverse way and turn each ciphertext into its original message. Since ( (a xor k) xor k) = a, we can just xor each ciphertext with the key to find its original message. If for each ciphertext, its original message is in the plaintexts array, then the key is valid. The result is the total number of valid keys we find.

Code

public class NetworkXOneTimePad
{
    // Performs xor between strings.
    // A common work around is to simply convert the strings
    // to integers. Do note that it introduces some corner cases:
    //  * Make sure to use 64 bit integers.
    //  * You have to convert the strings as base 2 numbers, not
    //    base 10 numbers ( String "101" in this problem represents
    //    number 5 and not 101).
    String xor(String x, String y)
    {
        String r = "";
        for (int i=0; i<x.length(); i++) {
            r += ( x.charAt(i) == y.charAt(i) ? '0' : '1' );
        }
        return r;
    }
    public int crack(String[] plaintexts, String[] ciphertexts)
    {
        int count = 0;
        for (String p: plaintexts) {
            // Find the key such that p xor key = ciphertexts[0].
            String key = xor(p, ciphertexts[0]);
            
            // Verify if for each ciphertext, its original
            // message using the key exists in plaintexts:
            boolean foundall = true;
            for (String cip : ciphertexts) {
                // Find the original message:
                String dec = xor(cip, key);
                
                boolean found = false;
                for (String x: plaintexts) {
                     found = (found || x.equals(dec) );
                }
                foundall = (foundall && found);
            }
            if (foundall) {
                count++;
            }
        }
        return count;
    }
}

NetworkXMessageRecovery
Used as: Division Two - Level Three:
Value 1000
Submission Rate 90 / 925 (9.73%)
Success Rate 30 / 90 (33.33%)
High Score Try_to_try for 879.55 points (10 mins 49 secs)
Average Score 586.67 (for 30 correct submissions)

Use exactly all the characters used in the input
It is sometimes vital to pay attention to details in the problem statement like the constraints. In this case, the original message did not have any repetitions and there is also a guarantee that at least one original messages of that kind will exist for the given subsequences.

Note that there may be many different characters in the input subsequences, and we need to use each of those characters because we want to find a message such that those strings are its subsequences.

There is also never a need to add any additional character. The task is to minimize the length of the original message and adding characters that did not originally exist in the subsequences does not help.

We also do not need to use the same character more than once. The original message will not have repetitions and the statement guarantees us that at least one message that follows those rules will exist.

As a conclusion, the shortest possible original message will consist of using all the characters that are available in the input subsequences.

The order
Although we know the characters that make part of the shortest result, we need to decide the order of the contents in a way that it is still a valid original message and is also the lexicographically-smallest solution.

Consider the example: {"acd", "bce"}. Each subsequence determines a mandatory order in which the characters must appear. Assuming acd to be a subsequence of the original message means that a will come before c and c before d in the original message. It could happen that two characters never appear in the same message fragment and thus there is no direct relationship. In that case, it is possible that indirect relationships do force a specific order. For example, in the case {"ad", "dc"}, c must come after d, and d must come after a, ergo c must come after a. In case there is not even an indirect relationship between two characters then placing any of them first is valid, but we will pick to place the smallest one first since we want to build the lexicographically-first solution.

The process of picking the order of a series of elements, when some elements must forcefully come after other elements is called a Topological sort. The graph is directed, and there is precedence between two characters if one comes before another in one of the fragments. There will never be cycles like, for example, {xy, yx}, because an original message with no repeated characters will always exist. There are at most 52 different characters, so we can use the simple topsort algorithm that just iteratively adds any character that does not have a pending requirement to the solution. In the special case when there are multiple characters with no specific requirements, we need to pick the lexicographically first one among them.

Code 

string recover(vector <string> messages)
{
    string topsort = "";
    int n = messages.size();
    while (true) {
        // Do a topsort
        const char INF = '{'; //A value higher than any the valid input
        
        char best = INF;
        // Find the character that is left to be added
        // has no requirements and is the
        // lexicographically smallest
        for (int i=0; i<n; i++) {
            if ( messages[i].size() != 0 ) {
                char ch = messages[i][0];  //Should a first
                                          // character in one of the messages
                for (int j=0; j<n; j++) {
                    // Confirm no requirement preceedes it.
                    for (int k=1; k<messages[j].size(); k++) {
                        if (messages[j][k] == ch) {
                            ch = INF;
                        }
                    }
                }
                best = min(best, ch);
            }
        }
        if ( best == INF) {
            // No more characters left
            break;
        }
        topsort += best;
        // Remove the character from all the lists of requirements:
        for (int i=0; i<n; i++) {
            if ( (messages[i].size() != 0) && (messages[i][0] == best) ) {
                messages[i] = messages[i].substr(1);
            }
        }
    }
    return topsort;
}

RowsOrdering
Used as: Division One - Level Two:
Value 500
Submission Rate 315 / 700 (45.00%)
Success Rate 115 / 315 (36.51%)
High Score Petr for 466.15 points (7 mins 46 secs)
Average Score 289.75 (for 115 correct submissions)

Let us first make sure to understand how the permutations will change the result. We want to find the minimum possible sum of the final positions of the given strings after they are sorted using the permutations. Each of the permutations represents an ordering. The permutation of the M columns will determine which column to check first when comparing. The permutations that belong to each column are basically rankings that function only for that column, which determines which character (number) comes first in comparison when we compare characters in that column. Note that the permutations for the columns may be different.

Let us first assume that the order of the columns has already been decided and is fixed. For example {0,1, 2, ...}. This means that the lowest-index columns will be compared first. What should we do about the rankings used by each column?

How does the ranking of each column affect the value we want to minimize? Let us think a simple case in which there is only one column. There is an example case like that: {a, b, c, d, e, f, g}, it should return 28. This is actually a simple case. No matter what ranking we decide, e.g., (a < b < c < d < e < f < g), the sum of the positions of the strings after sorting will always be: (1+2+3+4+5+6+7). This happens because all of the strings were different. How about a more special example like {a, b, c, c}? In this case the sum of the final positions can vary. With (a < b < c), the positions will be (1, 2, 3 and 3) and the sum will be (1 + 2 + 3 + 3). Note that this time the three is repeated. If we instead used: (c < a < b) the sum would be (2 + 3 + 1 + 1). Basically, the more times a string is repeated, the smaller the position we want to have for it.

An overall formula would be: sum( position * repetitions ) for each character in the column that ends in a position and appears repetitions times. We want to minimize this sum. After we found a list of frequencies/repetitions for each character in the column, we can sort this list in descending order and assign the lowest position to the character with the most repetitions, then the second-lowest position to the character with the second-most repetitions and etc. This greedy approach works and can be proven by a contradiction. Assume two characters a and b are repeated ra and rb times, respectively and are assigned to positions pa and pb. Then the sum will contain the following summands:

... + ra * pa + ... rb * pb + ...

Assume that (pa < pb) and (ra < rb). We can tell that: 

                  pb > pa
      (rb - ra) * pb > (rb - ra) * pa    //rb > ra
      
  ra * pa  + rb * pb > ra * pb + rb * pa

Which means that we could swap the positions of the characters in the ranking and get a smaller sum. This works for a single column and we will see it is true for any column in a setting with any number of columns. Let us move to a case {bb, bc, ac}. There are two columns, one column is {b, b, a} and the other is {b, c, c}. Assume for now that the ordering of the second column is fixed as (a < b < c < ...) (and the order of the columns is still fixed as {0, 1,...}). Let us see how the first column affects the positions:

We have strings {bb, bc, ac}. Assume first an ordering (b < a < c) . This means that all the strings that have a as a character in this column will come after those with b. The second column has the order (a < b < c). So the strings will be ordered in the following way: ..., bb, bc, ... ac. Remember that there are 50M different rows. So every possible string of M characters will exist. This means that there are 50M-1 strings that begin with b and an equal number of strings that begin with a. So there are some more strings between the ones that matter to us: ( ba, bb, bc, bd, ... bZ, aa, ab, ... Za, ...ZZ). We are assuming that Z is the last letter in the rankings of both the first and second column. Since there will be 50M-1 = 50 strings that begin with b, then we can tell the positions of the three strings: (bb = 2, bc = 3, ab = 52). In this case note that bb = 0*50 + 2, bc = 0*50 + 3 and ab = 1*50 + 2. This becomes very similar to numerical systems and it is like a base 50 number system with the exception that we decide the order of the columns and the values of the digits in each column.

Interestingly, we can determine a level of independence between the columns. Imagine that we picked the order (a < b < ... < Z) for the first column. Then the values would be: (bb = 1*50 + 2 , bc = 1*50 + 3, ab = 0*50 + 2). Note that the numbers provided by the second column have not changed: 2 + 3 + 2. We can calculate the best ordering for each column without considering the other columns.

Imagine the contents of a column were {a,a,b,a,c,c}, now it does not matter which column we are talking about. Only that it is not the last column (this exception will be dealt with later). Depending on the number of columns M and the number of column, we will multiply the digits of this column by a different power of 50. For example: 50, or 2500, or 5030. For now we will call this power P. The ordering that we pick for the characters in this column will affect the final result in the following way: Sum( position * repetitions * P ). Note that the rule that optimizes this shall be the same: assign the character with the most repetitions to the smallest position. Also note that in this case, the lowest position is 0 and not 1.

We did not consider the last column in the last case because in that case the first position number is 1. However, this can be changed. The reason the first position number is 1 is because the final positions are 1-indexed. If we assumed the positions to be 0-indexed, then the position numbers for the last column would also begin with 0. For example: {a, b, c} would get the sum (0 + 1 + 2) instead of (1 + 2 + 3). Note that to transform the final result from 0-indexed to 1-indexed, all that will take is to add up N, the total number of strings in the input. After moving to 0-indexed positions, the previous rule: P * Sum(position * repetitions) will apply to all columns, note that for the last column, P = 500 = 1.

Note that the optimal factor Sum(position * repetitions) for each column does not really depend on the column's assigned powers of 50. Therefore, it is possible to get the best of such value without considering what power of 50 we have. It only depends on what kinds of characters and repetitions exist in the column. This makes us ready to move to picking the best permutation of columns.

The best permutation of columns
The order of the columns actually determines the exponent of the power of 50 that will be multiplied to the factor = Sum(position * repetitions) for each column. For example, if we are talking about the first column, the result will have (factor * 50M-1) as one of its summands. If that was instead the last column, the result would have (factor * 1). Since the optimal factors for each column are already known, the problem becomes about picking the best position (exponent) possible for each of these factors.

Let us say that factor[i] is the calculated optimal factor for column i and that its position in the permutation of columns is position[i]. Then the final result will be: Sum( factor[i] * 50 M - position[i] - 1 , for each i). With a similar argument as the one we used before, we can see that it is best to assign the lowest factors to the highest column positions to minimize the sum of the positions.

Code

public int order(String[] rows)
{
    int M = rows[0].length();
    int N = rows.length;
    int[] columnFactors = new int[M];
    
    // Find the factors for each column:
    
    for (int i=0; i<M; i++) {
        // Count the frequencies
        // of each character in the column
        char[] ch = new char[N];  //extract the column
        for (int j=0; j<N; j++) {
            ch[j] = rows[j].charAt(i);
        }
        Arrays.sort(ch, 0, N);
        ArrayList<Integer> freqs = new ArrayList<Integer>();
        
        for (int j=0; j<N; j++) {
            if ( (j==0) || ( ch[j] != ch[j-1] ) ) {
                int c = 0;
                int k = j+1; 
                while ( (k < N) && (ch[j] == ch[k]) ) {
                    k++;
                }
                //Find the number of times this character appears
                // in the column.
                c = k - j;
                freqs.add(c);
            }
        }
        int factor = 0;
        Collections.sort(freqs);
        int dist = 0;
        // Assign the lowest ranks to the most frequent characters
        for (int j = freqs.size()-1; j>=0; j--) {
            int x = freqs.get(j);
            
            factor += x * dist;
            dist ++;
        }
        columnFactors[i] = factor;
    }
    // Sort the columnFactors
    Arrays.sort(columnFactors, 0, M);
    
    final int MOD = 1000000007;
    long res = 0;
    long p = 1;
    // Assign the left-most positions in the permutation
    // to the columns with the smallest columnFactors
    for (int i=M-1; i>=0; i--) {
        res += ( columnFactors[i] * p ) % MOD;
        
        p = (p * 50) % MOD;
    }
    res += N; //Convert result to 1-indexed
    return (int)(res % MOD); 
}

LexSmallestTour
Used as: Division One - Level Three:
Value 1000
Submission Rate 47 / 700 (6.71%)
Success Rate 6 / 47 (12.77%)
High Score rng_58 for 578.54 points (29 mins 10 secs)
Average Score 403.91 (for 6 correct submissions)

Regardless of the name, this problem is actually more similar to finding an Eulerian cycle than the travelling salesman problem. The only difference is that two consecutive roads must not be of the same type (the same character to represent the sold ware). Finding if there is an Eulerian cycle in an undirected graph depends on two simple properties:

All the vertices in the graph must have even degree.
All the vertices with degree larger than 1 (non-isolated) must form part of the same connected component.

The question is whether we can find similar properties that are as simple in order to find if the graph contains a cycle in which each two consecutive roads are of a different type. In fact, since this cycle must also be Eulerian, we can tell that those two properties are needed.

We certainly need to find more properties in relation to avoiding to use the same kind of road for two consecutive moves. Begin by noticing that the statement puts some emphasis on a special case: "Note that the final road he traverses is not considered to be consecutive to the first road he traverses.". How does that condition change the problem? Note that the rules for Eulerian cycles work in a way that you can begin the cycle at any arbitrary vertex. In the case of the problem that is not necessarily true. The cycle we want to find will sometimes necessarily begin at vertex 0. Because that would happen in case the last road and the first road are of the same type:

In this case, there is a tour for the salesman in the form (0 -> 2 -> 1 -> 0), but the last and first roads are of the same type. If we wanted to find the property needed to find if this tour exists, we would need to worry about vertex 0. If it was not for that condition, then it would probably be easier to find a property.

Let us call a tour such that every consecutive road used and also the last and first roads are of different type a perfect tour. We are currently supposing that finding a property to see if a perfect tour exists in a graph might be simpler than finding one for a normal tour (as described in the statement). If we knew the property for the perfect tour would we be able to somehow extend it so it works for normal tours as well? There is a way, imagine a normal tour existed and we wanted to make a perfect tour from it. In order to do it, we could a loop to vertex 0, and make it of a type not present originally in the graph. The idea is to add an intermediary road to use as the last road so that the repetition between the last and the first does no longer happen:

Note that the type of the road is $, that is because that is surely going to be different than the letters that are specified in the input. Loops though could complicate things in the future (for starters, the rule for Eulerian cycles does not originally consider them). We can avoid the loop by adding an intermediary vertex, this time we would need two roads, each with a different type. We can name the new vertex n. Note that we could even add another vertex n+1. Although we only need to add one vertex, there is a good reason to add two which will be explained later. Note that in both of the later cases, the Eulerian graph properties remain true. The degrees of the new vertices are even and the number of edges added to vertex 0 is even too. The two new vertices will be connected to vertex 0.

Note that if there is a normal tour in the original graph, there will be a perfect tour in the graph after we modify it that way. Simply because we added two or three roads that can be visited last or first and avoid the collision from the original path. More importantly, the converse is also true: If there is a perfect tour in the modified graph, there will be a normal tour in the original graph. To verify this, simply find the perfect tour then remove the cycle that is formed between vertex 0 and the new vertex (or the two new vertices in the second case).

Is there a perfect tour in a given graph?
We have just found out that knowing if there is a perfect tour in a given graph will also help us to know if there is a normal tour in the input graph. Let us find the properties that make a perfect tour possible.

First, take a look to vertex 0 in the left-most picture above. It has two edges of type A. Eventually, you will have to use the edges of each vertex, some as entrances to the vertex and some as exits. If you use an edge of type A to enter a vertex, you will have to use a different edge type to exit it. Note that this is where it is useful that we consider the last and the first road used as consecutive, because when we use the last road to enter a vertex, its equivalent exit time is the first road.

For every time we use a road of type A to enter a given vertex, we forcefully need to use a different road type as an exit and viceversa. From this it should follow that if a vertex has X edges of type A, then there should be at least X edges of a type different than A. This should happen for every vertex and for every road type: If the degree of a vertex is D, the number of edges of each certain type cannot exceed D/2. As we mentioned before, there are other two necessary conditions, the conditions used by Eulerian cycles.

What we will see now is that those three conditions (even degrees, a single connected component between all vertices with degree higher than 1 and no road type appears more than D/2 times for each vertex) are not only necessary, but also sufficient for there to be a perfect tour. There are many ways to do it, here is a quick one. Given the first two conditions, there will be at least one Eulerian cycle. If the Eulerian cycle in question had a instance of two consecutive roads of the same type, both roads would be connected to the same vertex. Since this vertex has two roads of the same type, then by the third condition it will have at least two roads of a different type. The Eulerian cycle will have outgoing and ingoing edges that visit the same vertex and have a different type. What is possible to do is to just swap the usage of these edges in the order they appear in the Eulerian cycles. For example:

In the figure, we cross the vertex i edges used in the first Eulerian cycle to get one in which the roads of type A are not consecutive. This can be done for every vertex that sees two consecutive roads of the same type.

Building the salesman tour
By first modifying the graph and using the properties described above to find if there is a perfect tour, we can tell if there is a valid salesman tour in the input graph. We still need to build the lexicographically-first one. In problems where we need to build a lexicographically-first solution and we know how to check if a solution exists in a setting we can use the following method:

For each step Salesman tour:
- Verify if it is possible to go to a specific city in this step. Pick the city that is the smallest (because we want the lexicographically-first sequence of cities) such that there exists a valid solution that uses this step.

Let us understand it in function of the current city in which we are and the type of the last picked road. We have already removed all used roads. There are a number of roads from the current city that we can use, but we should use only the smallest one that belongs to a valid solution. For each of those roads, we must verify if by using the road for this step we can travel through each of the other roads without repeating them and without using the same road type two consecutive times. In order to verify this, we can use our knowledge about perfect tours, but first build a graph that would yield a perfect tour if and only if it is possible to use the picked road in the resulting salesman tour.

Assume we are at vertex current and the last road was of type previousRoad. We can assume that at the beginning current = 0, and previousRoad = some character that does not exist. We want to check if it is possible to move to city i in the next step. Then we would like to make sure that it is possible to find a path such that:

The path begins by using the road between current and i.
That first road is of a type different than previousRoad.
The path visits each of the other roads exactly once and finishes in vertex 0.

In order to convert the problem to one of finding a perfect tour, we can remove the road in question, and instead somehow connect vertex 0 with i in a way that the last road in this new connection is of the same type as the road connecting current and i to ensure that the next road we pick is of a different type. Since we do not worry about the types of the roads between 0 and i in this special path, we would use special unused characters for the roads between 0 and i.

Again, the idea should be valid by adding only n and also if we add n+1 as well. Note that there is a perfect tour in the modified graph if and only if there is a valid path connecting vertex current (c) and 0 that follows the mentioned conditions. Given a perfect tour, you can simply remove the edges between 0, n, n+1 (if added) and i and re-add the road connecting c and i to find the correct path.

Code
We will repeat the process of finding an appropriate road |E| times (number of edges). For each of those |E| times, we need to check for at most |V| vertices (vertices connected to current). The check involves among other other things, verifying if there is only one connected component between vertices with more than 0 edges. Using a DFS, we can do that in O(|V| + |E|) time. In total we have O(|E|*|E|*|V|). There are at most V2 edges, so the total complexity is O(|V|5). It may appear large, but since there are at most 40 original cities and since we will usually find a correct road before using all V alternatives, it will run in time. The following c++ code requires less than 0.5 seconds in Topcoder's machines in its worst case.

In the explanation we pointed out that it is possible to add two special vertices instead of one vertex. The idea behind that is that when we add a single special vertex, we may have multiple roads connecting the same pair of vertex, this is no longer true if we add two special vertices. Since each pair of vertices has at most one edge, we can use the adjacency-matrix-esque format to represent the graph, which makes it simple to add and remove roads. It is possible to do it by adding only one special vertex, but the implementation is a little more complicated.

struct LexSmallestTour
{
    bool visited[52];
    
    void dfs(int x, const vector<string> & roads)
    {
        if (! visited[x]) {
            visited[x] = true;
            for (int i=roads.size() - 1; i>=0; i--) {
                if (roads[x][i] !='.' ) {
                    dfs(i, roads);
                }
            }
        }
    }
    
    bool perfectTour(const vector<string> & roads) 
    {
        int n = roads.size();
        fill (visited, visited+n, false );

        //* all degrees must be n.
        //* no ware should appear more than degree/2 times 
        // for a single vertex's edges.
        //
        for (int i=0; i<n; i++) {
            int degree = 0;
            map<char, int> count;
            for (int j=0; j<n; j++) {
                char r = roads[i][j];
                if (r != '.') {
                    count[r] ++;
                    degree ++;
                }
            }
            if ( degree % 2 != 0 ) {
                return false;
            }
            for (int j=0; j<n; j++) {
                if ( count[roads[i][j]] > degree/2 ) {
                    return false;
                }
            }
            if (degree == 0) {
                visited[i] = true;
            }
        }        
        
        // * All non-isolated vertices must be connected:
        int compos = 0;
        for (int i=0; i<n; i++) {
            if (! visited[i] ) {
                compos ++;
                dfs(i, roads);
            }
        }
        return (compos == 1 );
    }
    vector<int> determineLexFirstTour(vector<string> roads)
    {
        int n = roads.size();
        //add the n, (n+1), special vertices:
        for (int i=0; i<n; i++) {
            roads[i] += "..";
        }
        for (int i=0; i<2; i++) {
            roads.push_back(string(n+2,'.'));        
        }        
        // does a tour exist? 
        // a tour exists if and only if a "perfect" tour exists
        // in the following modified graph:
        // - Add an edge # between 0 and n.
        // - Add an edge $ between n and n+1
        // - Add an edge @ between n+1 and 0
        {
            vector<string> newg = roads;
            newg[0][n] = newg[n][0] = '#';
            newg[n][n+1] = newg[n+1][n] = '$';
            newg[n+1][0] = newg[0][n+1] = '@';
            if ( ! perfectTour(newg) ) {
                return vector<int>(0);
            }
        }

        // A tour exists, build the lex first one:
        vector<int> res(1, 0); //always start at 0.
        int current = 0; //current vertex
        char previousRoad = '?';

        while (true) {
            int i = 0;
            bool found = false;
            while ( (i < n) && ! found ){
                //try current -> i
                char r = roads[current][i];
                if ( (r != '.') && (r != previousRoad) ) {
                    vector<string> newg = roads;
                    //remove the road.
                    newg[current][i] = newg[i][current] = '.';    
                    //add a road 0 -> (n)
                    newg[n][0] = newg[0][n] = '$';
                    //add a road (n) -> (n+1)
                    newg[n][n+1] = newg[n+1][n] = '#';
                    //add a road (n+1) -> i with r as char.
                    newg[n+1][i] = newg[i][n+1] = r;
                    
                    if ( perfectTour(newg) ) {
                        found = true;
                        res.push_back(i);
                        // remove the road we just used.
                        roads[current][i] = roads[i][current] = '.';
                        current = i;
                        previousRoad = r;
                    }
                }
                i ++;
            }
            //Once we remove all the roads, we can stop.
            if (! found ) {
                break;
            }
        }
        return res;
    }
    vector <int> determineTour(vector <string> roads, vector <int> queries)
    {
        vector<int> tour = determineLexFirstTour(roads);
        
        if (tour.size() == 0) {
            return vector<int>(0);
        }
        
        int q = queries.size();
        vector<int> results( q );
        for (int i=0; i<q; i++) {
            results[i] = tour[ queries[i] ];
        }
        return results;
    }
};

Alternative solutions and additional comments.

A comment on "We repeat the process of finding an appropriate road |E| times." For each of those |E| times, the check can actually be done in O(|V|). Because to determine whether the edge (i,j) is the next piece of road, we only need to look at the neighborhood of node i: Choose the first j such that there is still a path from j to 0, using the remaining edges.
